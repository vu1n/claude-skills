  1. Turn 1 · agent · @surface

    Daemon-backed vs. self-contained file?

    Recommended: self-contained file. Archivability (a single HTML file you can commit, share, open in 2030) and trust (no global hooks installed silently). Daemon-backed pays its realtime polish in lifecycle complexity — we lived through that with lavish-axi.

    Alternatives: daemon-backed (lavish-axi-style; killed by dependency burden), hybrid daemon-when-available (too much surface area).

  2. Turn 2 · agent · @feedback-channel

    Question. Now that the composer lives in the artifact, how does it transmit your reply back to me? The composer can show what you typed, but the agent reads files (or the terminal), not browser memory. Three real options:

    • Copy-to-clipboard on Send. You paste once into the terminal (or into feedback.md) and I read from there. Works in every browser, no permission prompts, ~15 lines of JS.
    • File System Access API. One permission prompt per session, then Send writes directly to feedback.md next to the artifact. No paste step. Chrome/Edge only; Firefox partial; Safari none. ~60 lines of JS.
    • Auto-download a feedback file on Send. Works everywhere, but the user has to manage downloaded files and move them into the design directory. Worst UX of the three.

    Recommended: copy-to-clipboard now, FS Access API later. Copy gets us a working in-page composer this turn with minimal JS. If the paste step proves to be friction we'll feel it within a few real sessions, and we can upgrade to FS Access then — at which point we know the UX is worth the JS complexity. Doing FS Access first commits the artifact to ~60 lines of permission/file-handle plumbing forever, against a friction we haven't measured.

    Alternatives considered. Auto-download is strictly worse than either real option. FS Access now is a defensible call if you'd rather pay the JS cost once than carry the paste friction in every session.
