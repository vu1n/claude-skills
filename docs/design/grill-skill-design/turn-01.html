Surface

What does the agent grill against? Daemon-backed (chat panel, live reload, click-anywhere annotation) vs. a self-contained HTML file the user opens in any browser. Foundational choice — everything downstream depends on this.

Versioning

One mutating artifact updated in place vs. a new turn-NN.html snapshot per turn. Drives diff-ability, archive value, and disk cost.

Feedback channel

How does the user reply? Terminal-only, in-page composer with a copy button, sidecar markdown file, or some combination. Determines where notes persist and how much page-internal JS the artifact needs.

Annotation precision

How precisely can the user point at this vs. that? Element-pickers with DOM hit-detection, agent-declared anchor refs, or no annotation at all (free-form prose). Trades JS complexity against precision.

Decision recording

Where do resolved decisions land and what shape do they take? Inline in artifact, sidecar markdown, structured format (.facts-style), or ADRs. Lifecycle tags, ecosystem compatibility stubs.

Turn 1 · @surface — daemon-backed vs. self-contained file?

Question. What does the agent grill against — a daemon-backed surface (chat panel + live reload + click-anywhere annotation, hosted by a local server) or a self-contained HTML file the user opens in any browser?

Recommended: self-contained file. Two things drive this. First, archivability — a single HTML file you can commit, share via email, open in 2030 with no setup is a better design record than a transient chrome session that disappears when the daemon dies. Second, trust — daemons that auto-install global hooks (we just lived through this with lavish-axi) violate user expectations and get tools uninstalled. The self-contained file pays for these wins with copy-paste friction on annotations; we'll resolve that in @annotation-precision and @feedback-channel.

Alternatives considered.

  • Daemon-backed (lavish-axi-style): realtime push, integrated chat, click-anywhere precision. Killed by dependency burden — install steps, hook lifecycle, version coupling between CLI and host.
  • Hybrid: self-contained file, but with an optional daemon that augments it when running. Too much surface area; ends up shipping the daemon's complexity to anyone who wants the "good" version. Skipped.

Resolving this question first because every other branch (versioning shape, feedback channel, annotation mechanism) is shaped by whether we have a runtime to lean on.
